The Use Case

A customer walks up to the kiosk. Their face is scanned and matched to their Odoo record. From that moment, the AI concierge has everything a seasoned salesperson would know — purchase history, open orders, loyalty status, outstanding issues — and can have a fully informed, natural conversation to assist, upsell, and serve them.

Phase 1 delivered the fundamental Odoo CRUD patterns — 17 scripts covering leads, orders, invoices, inventory, and events. Phase 2 is about making the concierge genuinely intelligent: composite queries, cross-system analytics, and the ability to take meaningful action in real time. The centerpiece of Phase 2 is the Tier 3.5 Data Normalization Layer — the middleware that makes the concierge platform-agnostic and analytics-ready.

Every script in this document maps to one or more stages in this loop. Tier 3.5 is what makes Stage 3 (Context Load) scalable across data sources — it is the bridge between raw API data and the concierge's reasoning layer.

TIER 3.5 — Data Normalization Layer  ★ NEW

Data source: ANY — Odoo, Salesforce, custom, homegrown • Recipe prefix: normalize_* • is_read_only=1 • is_chainable=1

What Tier 3.5 Is

Tier 3.5 is the ETL middleware of the cookbook. Each normalizer recipe calls one or more Tier 1/2 Get scripts, maps source-specific field names and data structures into a canonical Common Data Model (CDM), and returns a clean, typed Python dict or list that Tiers 4–6 can consume without knowing anything about the data source.

No new tables are needed. Tier 3.5 recipes are pure Python — on-the-fly ETL inside the kitchen sandbox. They read from Tier 1/2 scripts (or from Redis cache), transform, and return. Redis caches their output so subsequent analysis calls skip the round-trip to the source system.

Why This Layer Matters

Platform agnostic — swap Odoo for Salesforce tomorrow, write one new normalizer, and every Tier 4–6 analytics script keeps working unchanged.

LLM-optimized output — the concierge's reasoning layer gets clean, minimal, semantically consistent data. No raw API payloads, no null fields, no Odoo-specific field names like x_studio_field_abc.

Redis integration point — Tier 3.5 is where caching happens. Normalized results are stored with a TTL. Tier 4–6 scripts read from cache first, making repeated analysis in a single session nearly instantaneous.

Self-describing contracts — each normalizer declares its output schema as a CDM type, making recipe dependencies explicit and enabling the kitchen to validate chain integrity before execution.

Foundation for the wow factor — real-time what-if analysis is only possible when the normalization step is cached. 'What if I increase safety stock by 20 units?' reruns in <200ms because only the simulation step changes, not the data pull.

Normalizer Recipe Conventions

All normalizers: is_read_only=1, is_chainable=1 (they can be called by upper-tier chains)

vars contract: source_system defaults to 'odoo'; caller can override to 'salesforce' etc.

Return envelope: normalizers still return the standard {status, http_status, data, user_message, system_message} envelope — the CDM object lives in data.

Cache key pattern: norm:{recipe_name}:{sha256(sorted_vars_json)[:8]} — Jason's kitchen will compute and store this automatically when cache_ttl_seconds > 0.

Dependency declaration: describe depends_on in the recipe description field for now. A cookbook_recipe_dependency table is a planned schema addition (Jason's call on timing).

REDIS CACHING LAYER — Architecture

Redis sits between Tier 3.5 normalizers and the Tier 4–6 analyzers. The kitchen engine (Jason's domain) manages cache read/write transparently — recipe authors set cache_ttl_seconds and the engine handles the rest. The result: a customer kiosk session that runs 10 analytics scripts feels like it's running 2, because 8 of them hit cache.

Cache Invalidation Strategy

Write actions: Tier 3 Action scripts (create_*, update_*, apply_*) call kitchen.cache.invalidate(key_pattern) after a successful write.

Simulation: Simulation keys (sim:*) are session-scoped and never written back to the source system. They exist only for what-if analysis.

Implementation: Jason wires explicit invalidation calls into cook.py — recipe authors signal intent via a declared invalidates_cache: ['key_pattern'] field in the recipe metadata.

Fallback: If Redis is unavailable, the kitchen falls back to direct source calls transparently — degraded performance, not a failure.

TIER 6 — Self-Correcting Models & Backtesting  (the learning layer)

Data source: Historical CDM + prior Tier 5 outputs • Recipe prefix: backtest_*, calibrate_*, score_*

Tier 6 closes the loop. It takes past Tier 5 predictions, compares them to what actually happened, quantifies the error, and adjusts model weights or thresholds stored in cookbook_system_config. The system gets smarter over time without requiring a model retrain or a code deploy.

A store manager stands at the kiosk and asks: 'What would happen to our stockout risk if we increased safety stock on our top 5 sellers by 20 units each?'

What Makes This Impressive

Steps 1–3 run once per session and are cached. Steps 4–6 are sub-second on every what-if iteration.

No data is written to Odoo during the simulation. The sim: key in Redis is session-scoped and discarded at session end.

The manager can iterate freely: 'What about +40 units? What if we only do the top 2? What if we move stock from Location A to B?' Each iteration reruns in milliseconds.

This is the same loop Tier 6 will use for backtesting — 'how accurate was last week's prediction?' runs against the same analyze_inventory_gaps script with historical data.

The Key Insight for Phase 2

Phase 1 scripts are building blocks. Phase 2 is about the layer above them. Tier 3.5 is the structural bet: invest in normalization once, and every analytics script you write from that point forward works across any data source you add — Odoo today, Salesforce next quarter, custom POS after that.

The ‘wow factor’ is not a single impressive script. It is a system that can answer ‘what if?’ questions in real time, learn from whether its answers were right, and get better without a redeploy. That is what Tiers 3.5 through 6 deliver together.

What it unlocks

Tier model

cookbook_recipe.tier (V105)

Lets the admin UI, the LLM tool manifest, and the dependency graph filter recipes by tier (tier1 / tier2 / tier3 / tier3.5 / tier4 / tier5 / tier6).

CDM registry

cookbook_cdm_type + cookbook_cdm_field (V105)

Per-CDM-type field definitions (CommonCustomerProfile, CommonTransaction, etc.) live as rows. Recipes declare cdm_output_type / cdm_input_types_csv so chains type-check at registration time.

Dependency graph

cookbook_recipe_dependency (V105)

Normalizers depend on Gets, Analyzers depend on Normalizers. Junction table replaces free-text dependency notes and enables future auto-invalidation propagation.

Cache keys

kitchen/cache.py + cookbook_recipe.cache_scope

New norm:{recipe}:{hash} key (args-only, shared across the session) and sim:{recipe}:{session_id} key (per-session overlay for what-if). Existing kitchen:recipe:* key stays for tenant-scoped recipes.

Invalidation

cookbook_recipe.invalidates_cache_csv + cook.py hook

Tier 3 writes declare the Redis key patterns they invalidate. After a successful sandbox run, cook.py invokes invalidate_patterns() (SCAN-based) so downstream analyzers see fresh data on the next call.

LLM tools

/v1/tools ?tier= filter + /v1/cdm-types endpoint + llm_gateway wiring

LLM only sees tier1-tier3 by default (avoids the LLM trying to call an analyzer directly). CDM type list joins the system prompt so the model understands what shapes flow between tiers. Tool-call interceptor invokes /v1/cook with TOOL_LOOP_MAX_STEPS cap.

How existing Phase 1 recipes get updated

PR-1 includes a backfill script that annotates every existing odoo recipe under hyve-blackbox/hyve-kitchen/recipes/odoo/ with the new fields. The rules are deterministic, based on filename verb: odoo_get_* and odoo_search_* become tier=‘tier1’ (CRM-module recipes) or ‘tier2’ (Product/Inventory module); odoo_create_*, odoo_update_*, odoo_delete_*, odoo_confirm_*, odoo_cancel_*, odoo_post_*, odoo_apply_*, odoo_validate_* become tier=‘tier3’. cache_scope defaults to ‘tenant’ (no change from current behavior); cache_ttl_seconds stays whatever the recipe author set. invalidates_cache_csv is left NULL on all existing recipes — we’ll fill it in as Phase 2 normalizers land that the writes need to invalidate. No re-test of Phase 1 recipes is required: the new columns all default safely.

Three-PR sequencing

PR-1 — Schema + cache plumbing. V105 migration (tier, cache_scope, invalidates_cache_csv, cdm_output_type, cdm_input_types_csv columns; new tables cookbook_cdm_type, cookbook_cdm_field, cookbook_recipe_dependency). cache.py adds build_norm_cache_key / build_sim_cache_key / invalidate_patterns. cook.py wires the invalidation hook after every successful Tier 3 write and accepts a simulation_session_id field. Backfill script annotates existing recipes. Tyler & Allan can begin authoring Phase 2 recipes against the new shape after PR-1 lands.

PR-2 — Registry & tool manifest enrichment. registry.py exposes the new columns; /v1/tools gains the ?tier= filter and per-tool metadata.{tier, chainable, cdm_output, cdm_inputs}; new /v1/cdm-types endpoint returns the CDM registry for prompts & docs. CONVENTIONS.md updated with the new mandatory fields per tier. Unblocks the admin UI to render tier badges and the dependency graph.

PR-3 — LLM gateway wiring. Drops the existing gateway_integration/kitchen_router.py into hyve-llm-gateway/llm_gateway.py (lifespan boot + chat-completion request loop). Tier-filtered tools[] injection + CDM system-prompt section + tool-call interceptor capped at TOOL_LOOP_MAX_STEPS=4. Refresh-tools webhook fires on POST /v1/recipes so newly inserted recipes appear in the LLM’s tool list within one chat turn instead of after a 5-minute timer. This is the PR that closes the LLM → kitchen loop end-to-end.
