Phase 2 Analytics Recipe Author Guide

For Tyler (Tier 3.5 normalizers) and Allan (Tier 4 / 5 / 6 analyzers, simulators, calibrators).

Read this first

This guide is the working companion to hyve_phase2_brainstorm_v2.docx. The brainstorm document explains why the tier model exists and what each layer is for. This document explains how to author a recipe that lands in the right tier, declares the right cache scope, type-checks against the CDM, and chains correctly with its dependencies.

Read it once end-to-end, then keep it open while you write your first three recipes. By the third, the pattern is automatic.

All Phase 2 recipes live under:

hyve-blackbox/hyve-kitchen/recipes/<system>/<module>/<recipe_name>.sql

with the exception of Tier 3.5 normalizers, which live under:

hyve-blackbox/hyve-kitchen/recipes/_normalizers/<cdm_type>/<recipe_name>.sql

because they are system-agnostic by design. A normalizer that maps Odoo crm.lead into CommonCustomerProfile lives next to one that will map Salesforce Account into the same CommonCustomerProfile, so analyzers further up the stack do not care which source produced the dict.

The 30-second authoring loop

Copy _templates/recipe_insert.template.sql to the right tier folder.

Rename the file to <recipe_name>.sql; that string must equal cookbook_recipe.name.

Fill in the 8 standard sections plus the 5 new Phase 2 fields (tier, cache_scope, invalidates_cache_csv, cdm_output_type, cdm_input_types_csv).

If you depend on other recipes (almost every Tier 3.5 + Tier 4 recipe does), add one cookbook_recipe_dependency INSERT per upstream recipe.

Apply to your test DB: docker exec -i hyve-postgres psql -1 -U hyve_admin -d hyve_kitchen_test < your_recipe.sql

Verify with _verify/manifest.sql; the recipe should show up tagged with the right tier.

If your recipe is Tier 1, 2, or 3, it appears in the LLM tool manifest automatically (PR-3). If it is Tier 3.5 or higher, it is only callable via chains and is not directly LLM-exposed.

Cache scope semantics

'tenant' (default; T1/T2/T3 reads, T3.5/T4 standard reads)

Key: kitchen:recipe:{name}:result:{sha256(args + subscription_id + character_id)}. Per-tenant isolation; one customer's analytics never leak into another's session. Cache TTL set by cache_ttl_seconds on the recipe row.

'session' (T5 simulators, simulation-mode reads)

Key: sim:{recipe_name}:{simulation_session_id}. The cook request body carries a simulation_session_id; cook.py reads/writes this key when the recipe's cache_scope is 'session' OR when the request explicitly forces simulation mode. Used so what-if iterations write to a session-local overlay that never touches Odoo and gets discarded at session end.

'global' (rare; T2 product catalog, system-wide config)

Key: norm:{recipe_name}:{sha256(args)} -- args only, shared across every session. Use this only when the cached output is truly tenant-agnostic (a public product spec, a published tax rate). Misuse is a privacy bug.

Consumes CDM lists from one or more Tier 3.5 normalizers. is_read_only=1, is_chainable=1, cache_scope='tenant'. Declare every CDM input type the recipe consumes in cdm_input_types_csv -- this is how PR-2's tool manifest tells the LLM 'this recipe needs a CommonTransaction list to run.'

Template C -- Tier 5 simulator

Reads a CDM list from cache, applies an in-memory delta, writes the modified list back to sim:{recipe}:{session_id}. Never touches the source system. is_read_only=1 because it does not write to source; cache_scope='session' so the overlay is session-bound.

Sample recipe: analyze_spend_trend

Shipped at hyve-blackbox/hyve-kitchen/recipes/odoo/analytics/analyze_spend_trend.sql. Apply it to a V105 database, point a recipe-chain at it, and you have a working Tier 4 analyzer. Copy it as the starting point for analyze_inventory_gaps, analyze_churn_signals, and analyze_customer_ltv -- the shape is identical, only the math changes.

What the sample demonstrates

Every Phase 2 field set correctly (tier='tier4', cache_scope='tenant', cdm_output_type='SpendTrend', cdm_input_types_csv='CommonTransaction', cache_ttl_seconds=120, is_chainable=1).

One cookbook_recipe_dependency row pointing at normalize_transaction_history (which Tyler ships in his sprint).

Standard 8-section file shape (systems / modules / auth / globals / recipe + version CTE / vars / outputs / verify block).

Python body that consumes a CommonTransaction list, computes weekly aggregations and a prior-period delta, returns a SpendTrend dict matching the declared cdm_output_type.

Returns the standard envelope (status / http_status / data / user_message / system_message).

Validates entirely in Python before touching anything -- never raises out of run().

Common mistakes to avoid

Tier 4 recipe declares cdm_input_types_csv but the upstream normalizer doesn't exist yet. Fix: add the cookbook_recipe_dependency row anyway; the manifest will warn until the upstream lands and that's the point.

Cache TTL too long for a write-adjacent normalizer. A normalize_inventory_snapshot with TTL > 60s will serve stale stock data the instant any reserve_item_for_customer runs. Set Tier 3 writes' invalidates_cache_csv correctly OR keep the TTL very short.

Tier 5 simulator forgets to set cache_scope='session'. Result: simulation output lands in the tenant cache and bleeds into the next non-simulation analytics call. cache_scope='session' isolates the overlay to the session_id.

Python body raises out of run(). The kitchen sandbox reports 'sandbox_error' with no context. Wrap every IO call in try/except; return the standard error envelopes (MISSING_INPUT, VALIDATION_ERROR, AUTH_FAILED, UNKNOWN_ERROR) instead.

Forgetting cookbook_recipe_dependency rows. The recipe still runs, but the admin GUI's dependency graph (PR-2) cannot render the lineage and the future auto-invalidation traversal cannot follow the chain.

Testing your recipe

Three checks before opening a PR:

1. Apply the SQL clean: psql -1 ensures the file rolls back fully on any error. Apply against a fresh hyve_kitchen_test DB.

2. Manifest shows it: SELECT name, tier, cache_scope, cdm_output_type FROM cookbook_recipe WHERE name='your_recipe';

3. Invoke via /v1/cook: build a curl with the right vars; for Tier 4+ that depends on cached normalizer data, first hit /v1/cook for the upstream normalizer so the cache is warm.

If you need to inspect cache state, use:

docker exec -it hyve-redis redis-cli KEYS 'norm:*'

docker exec -it hyve-redis redis-cli GET 'norm:customer_profile:abc123...'
